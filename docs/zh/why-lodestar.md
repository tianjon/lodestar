# 为什么需要 Lodestar

## 真正的问题是定向衰减，而不是记忆丢失

自回归 coding agent 天然受近因影响：每条新消息都会强烈牵引下一次输出。对话越长，这种牵引越会累积。agent 会悄悄开始优化“最新指令”，而不是最初启动这次工作的目标；原始目标会一轮一轮衰减。

真实 agent harness 里有四种机制会反复切断这条线：

- **上下文压缩**会把对话有损总结后继续。
- **新会话**几乎从空上下文开始，只会自动加载少量静态文件。
- **任务技能**，比如 TDD、debugging、review，会引入自己的流程，可能拉走注意力。
- **子代理**通常以全新上下文运行，不天然继承父任务的目标方向。

危险的失败模式不是忘记某个事实，而是**静默滑移**：agent 依然流畅、自信、忙碌，但已经瞄准了错误目标。直到大量工作完成后，人们才发现方向错了。

## 这是控制问题，不是搜索问题

记忆库或向量检索可以找回**事实**。定向是另一件事：它要稳定保存“我们现在到底要做什么，以及下一步行动是否服务这个目标”。你不能只靠搜索回到正轨；你需要一个可以对齐的设定点。Lodestar 是这个设定点，而不是另一个索引。

## Lodestar 补上的东西

Lodestar 提供一个小而持久的项目本地**锚点**，用来穿过那些会擦除对话的机制：

- 当前**模式**：explore / clarify / decide / execute / review；
- 一个活跃**目标**，用一句话写清楚；
- 可观察的**完成定义**；
- 明确的**边界**：这一阶段什么不做；
- 项目的**领域语言**；
- 开放**差距**，带证据和置信度；
- **决策**，包含替代方案和理由；
- **下一步行动**。

## 立场：真实，而不是顺从

Lodestar 在源头记录用户要求，然后诚实地把它和当前实践对照。它既不奉承用户，也不盲从资料。要求和实践冲突时，它可以基于证据挑战目标本身，也可以提出突破二者的第三条路径。原则是突破经验限制，而不是在错误边界里盲目优化。

## 它不是什么

Lodestar 不是通用记忆库，不是向量搜索，不是任务技能的替代品，也不试图记住每一句话。它保存的是定向，而不是聊天记录。

## 诚实的适用范围

Lodestar 的价值是真实的，但有条件。它最适合长程、多会话、跨 harness 的工作，因为这些场景里意图确实容易衰减。对于目标已经清晰、单会话就能完成的小任务，它可能只是额外开销。半维护的锚点甚至比没有更糟，因为过期目标会带着书面记录的权威误导后续工作；这也是为什么维护纪律不是装饰，而是系统的一部分。

## 期望状态

当 Lodestar 有效时，一个恢复后的会话应该能用一句话说清楚：

> 我们正在推进这个目标；成功意味着这个；这些事情不在范围内；当前差距是这个；下一步最有用的行动是这个。
