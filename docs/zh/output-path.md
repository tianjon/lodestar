# Lodestar 如何影响 Agent 产出

## 记录不是目的；改变下一步行动才是目的

对一个定向系统来说，唯一重要的测试是：保存下来的状态是否改变了下一次回答、工具调用、review、计划或代码修改。不会改变下一步行动的状态是负担；而看起来很认真却不改变行动的负担，比没有更糟。

## 定向管线

```text
用户指令
  -> 领域解析        （术语、对象、边界）
  -> 目标 / 完成定义 / 边界
  -> 差距            （要求 vs 现实，带证据和置信度）
  -> 决策            （选择的路径、替代方案、理由）
  -> 下一步行动
  -> 产出框架        （回答或修改围绕目标构建）
  -> 状态更新        （闭环；喂给下一轮）
```

这个 loop 是有意闭合的：每一轮都让状态比上一轮更清楚一点。

## 状态到产出的四个入口

1. `anchor.md` 告诉 agent：**产出必须服务什么**。
2. `domain.md` 提供解释任务所需的**词汇和边界**。
3. `state.md` 告诉 agent：**哪些已经为真，哪些差距仍然开放**。
4. hook 在 agent 行动前**注入这些上下文**，让它们在决策时到场，而不是靠模型自己想起来。

## 好的 Lodestar 对齐产出是什么样

- 它回答**活跃目标**，而不只是回应最近一句话。
- 它尊重**边界**。
- 它使用项目自己的**领域语言**，而不是通用模板。
- 它缩小一个命名 GAP，或推进下一步行动。
- 它给下一次会话留下比进入时更好的状态。

## 它要防止的反模式

agent 自信且高效地回答了最近那个子问题，但真正的项目目标没有被推进。这就是单次产出层面的静默滑移。上面的管线存在的目的，是在答案写出之前捕捉它，而不是事后复盘才发现。
